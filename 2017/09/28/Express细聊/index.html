Express细聊

2017-09-28 PV:

如果说到Node.js那么一定就会说到Express

Express.js如它的官网的介绍：快速、可选式、轻量的Web框架（Fast, unopinionated, minimalist web framework）。

如果说要熟悉一个框架或者某一门语言，个人觉得最快的方式就是去阅读它的官方文档，最好是原始的英文不要翻译的那种。本文也是基于Express最新的官方文档来对Express进行一个比较细致的学习。

Node.js的HTTP模块

Node.js最开始推出来的时候，作者对它的定义就是一个高性能的非阻塞I/O的web服务器，所以很明显原生的Node.js就提供了很完善的http相关的接口来帮助你快速的搭建你自己的一个web服务，所以说Node.js本身是有http模块能够便捷的提供web服务；

const http = require('http');

const hostname = '127.0.0.1';
const port = 3000;

const server = http.createServer((req, res) => {
  res.statusCode = 200;
  res.setHeader('Content-Type', 'text/plain');
  res.end('Hello World\n');
});

server.listen(port, hostname, () => {
  console.log(`Server running at         http://${hostname}:${port}/`);
});

但是当我们需要一个路由，需要根据不同的http method来进行不同的response，这个时候http 模块的短板显露无疑，这个时候Express应运而生。

基于Express搭建一个Web服务

简单来说Express就是在Node.js的基础上进行了一层封装帮助开发者更加简单的搭建Web服务；

app.listen = function listen() {
  var server = http.createServer(this);
  return server.listen.apply(server, arguments);
};

以上就是Express源码中/lib/application.js中对node.js的http模块的createServer方法的封装。

那么利用Express能多快就能搭建一个web服务呢？答案是4行代码；

const express = require('express');       // 引入express
const app = new express();                // 创建Express对象
app.get('/', (req, res, next) => res.send('hello world!')); // 响应请求
app.listen(8080);                         // 对端口监听

当然一个稳定且安全的web服务一定不是这么简单的就完了，我们需要路由，需要拦截，需要错误处理，需要静态资源的访问，巴拉巴拉各种需要；

冷静点，Express通过一个巨有效的机制帮你解决以上所有的需求；

下面较为隆重的介绍Express的middleware机制：

Middleware

Express is a routing and middleware web framework that has minimal functionality of its own官网有这么一段原话，说Express是一个自带有十分少功能的路由和中间件的框架；这就意味着如果你要搭建一整套完善的线上能用的web框架的时候，你需要安装一堆的中间件，并且做好路由的转发。

事实上，中间件就是一个函数，能够访问到请求中的数据，访问的到返回的数据，并且能够调用栈当中的下一个中间件函数。

那么中间件是干什么的呢?

之前说过Express本身是不带任何什么cookie处理呀，日志存储之类的功能的，所有的常用的Web服务所需要的功能都是通过一个个中间件来处理，于是乎一个中间件函数具备以下几点能力：

  • 执行一些代码
  • 对req请求对象和res返回对象进行修改
  • 直接结束当前request-response循环
  • 调用栈当中的下一个中间件函数

如果当前的中间件函数没有结束当前的请求，response的过程的话，就必须调用下一个中间件函数，不然这个请求就被hanging住了。

从框架的层次结构上看一个Express应用包含以下几类中间件：

  1. 应用级别的中间件；
  2. 路由级别的中间件；
  3. 错误处理中间件；
  4. 内建中间件；
  5. 第三方中间件；

应用级别的中间件

通过app.use和app.METHOD对web应用对象添加中间件。这里METHOD对应的就是HTTP当中的请求的方法比如get/put/post之类的；

var app = express()

app.use((req, res, next) => {
  console.log('Time:', Date.now())
  next()
})；

app.get('/user/:id', (req, res, next) => {
   res.send('USER'); 
});

可以看到通过调用use方法挂载了一个匿名函数，这个匿名函数就是一个中间件，里面打印了调用这个中间件的时间，然后调用下一个中间件函数；第二个挂载的中间件第一个参数是一个RESTful的路径，当请求匹配到这个路径的时候，将调用这个中间件函数，可以看到这个中间件函数当中执行了一个res.send(‘USER’)返回一个200的response并且其body里面为一个字符串USER;

这里就会有一个疑问了，这些中间件在栈中的顺序是怎么样的？

带着同样的疑问我写了一个简单的例子：

// 1
app.use((req, res, next) => {
    console.log(`first no path`);
    next();
});

// 2
app.use('/', (req, res, next) => {
    console.log('first / path');
    next();
});

// 3
app.use('/', (req, res, next) => {
    console.log('second / path');
    next();
});

// 4
app.use('/a', (req, res, next) => {
    console.log('/a path');
    next();
});


// 5
app.use('/:id', (req, res , next) => {
    console.log('/ path with param');
    next();
});

// 6
app.use((req, res, next) => {
    console.log('no path!');
    res.send('hello middleware!');
});

// 7
app.use((req, res, next) => {
      console.log('last no path') ;
    res.send('Last middleware');
});

然后curl http://localhost:8080/1

最后请求的结果：

first no path
first / path
second / path
/ path with param
no path!

可以发现express是基于挂载的顺序入栈， 这也意味着express对所有的中间件函数都是一视同仁，但是当路径没有匹配的时候，则跳过该函数。并且可以注意到当调用到第6个函数的时候，直接跳出了当前的中间件栈，向页面返回了一段hello middleware!，这也是middleware具有的一个很重要的退出request-response cycle的功能；

在v4.11.0之后被废弃了的app.param()方法会打破按顺序加载中间件的规则，app.param当中声明一个数组，当中存放了一些参数关键字，当参数字段命中了的时候会在该路径的回调执行之前执行app.param当中声明的回调函数。In What Order Does My Express.js Middleware Execute?

路由级别中间件

可以理解路由级别跟应用级别的中间件行为是一致的，除了它是被绑定到express.Router()对象上的；

路由级别的中间件其实就是在应用级别的中间件下的分支，举个栗子：

const app = express();
const router = express.Router();

router.use((req, res, next) => {
    console.log(`Time: ${Data.now()}`);
      next();
});

router.use('/b', (req, res, next) => {
       res.send('hello router'); 
});

app.use('/a', router);

app.listen(8080);

可以看出当命中了/a/b的时候，应用级路由会调用router级别的中间件，在router级别中间件当中会先打印一下当前的时间，再在命中了/a/b路径之后，向网页发送个hello router；

错误处理中间件、内建中间件

错误处理中间件类似于应用的不带路径的中间件函数，就是参数的数量是4个，第一个用于接受错误对象

app.use((err, req, res, next) => {
      console.error(err.stack);
      res.status(500).send('Someting Broke');
});

Express内建的中间件就一个express.static(root, [options])方法，用于提供静态资源的访问服务。

第三方中间件

Debug

Expres基于debug模块来提供debug功能。

这里的Debug功能很类似于写js中用到最多的console.log方法，但是不同于console.log你不需要在生产环境下去全局搜索，然后再注释掉这些console.log，你只需要在开发的时候打开debug功能，在上线的时候不打开就行了。

在Express开发中，如果你需要看到所有的内部的log信息，只需要一行

DEBUG=express:* node index.js

通过设置DEBUG环境变量为express:*就行了；

Express进程管理

当在生产环境使用Express的时候，使用一个进程管理来监听Express的进程是非常有必要的。简单的带来以下好处：

  • 自动重启崩溃的Express进程；
  • 能够监听运行时Express进程的性能和资源消耗状况
  • 动态的修改配置以提高应用性能
  • 集群控制

一个进程管理类似于一个应用Server能够作为一个应用的容器用来快速部署，提供高可用，并且让你能够在应用运行时管理应用；

常用的一些进程管理工具：PM2，forever；

PM2

自带负载均衡，这个有点厉害；

能够快速的帮你通过cluster的模式启动四个进程。还能通过pm2 scale [app-name] 10的方式快速扩展你的cluster到10个进程；
