Express细聊

2017-09-28 PV:

如果说到Node.js那么一定就会说到Express

Express.js如它的官网的介绍：快速、可选式、轻量的Web框架（Fast, unopinionated, minimalist web framework）。

如果说要熟悉一个框架或者某一门语言，个人觉得最快的方式就是去阅读它的官方文档，最好是原始的英文不要翻译的那种。本文也是基于Express最新的官方文档来对Express进行一个比较细致的学习。

Node.js的HTTP模块

Node.js最开始推出来的时候，作者对它的定义就是一个高性能的非阻塞I/O的web服务器，所以很明显原生的Node.js就提供了很完善的http相关的接口来帮助你快速的搭建你自己的一个web服务，所以说Node.js本身是有http模块能够便捷的提供web服务；

但是当我们需要一个路由，需要根据不同的http method来进行不同的response，这个时候http 模块的短板显露无疑，这个时候Express应运而生。

下面较为隆重的介绍Express的middleware机制：

Middleware

Express is a routing and middleware web framework that has minimal functionality of its own官网有这么一段原话，说Express是一个自带有十分少功能的路由和中间件的框架；这就意味着如果你要搭建一整套完善的线上能用的web框架的时候，你需要安装一堆的中间件，并且做好路由的转发。

事实上，中间件就是一个函数，能够访问到请求中的数据，访问的到返回的数据，并且能够调用栈当中的下一个中间件函数。

那么中间件是干什么的呢?

之前说过Express本身是不带任何什么cookie处理呀，日志存储之类的功能的，所有的常用的Web服务所需要的功能都是通过一个个中间件来处理，于是乎一个中间件函数具备以下几点能力：

  • 执行一些代码
  • 对req请求对象和res返回对象进行修改
  • 直接结束当前request-response循环
  • 调用栈当中的下一个中间件函数

如果当前的中间件函数没有结束当前的请求，response的过程的话，就必须调用下一个中间件函数，不然这个请求就被hanging住了。

从框架的层次结构上看一个Express应用包含以下几类中间件：

  1. 应用级别的中间件；
  2. 路由级别的中间件；
  3. 错误处理中间件；
  4. 内建中间件；
  5. 第三方中间件；

应用级别的中间件

通过app.use和app.METHOD对web应用对象添加中间件。这里METHOD对应的就是HTTP当中的请求的方法比如get/put/post之类的；

可以看到通过调用use方法挂载了一个匿名函数，这个匿名函数就是一个中间件，里面打印了调用这个中间件的时间，然后调用下一个中间件函数；第二个挂载的中间件第一个参数是一个RESTful的路径，当请求匹配到这个路径的时候，将调用这个中间件函数，可以看到这个中间件函数当中执行了一个res.send('USER')返回一个200的response并且其body里面为一个字符串USER;

Debug

Expres基于debug模块来提供debug功能。

这里的Debug功能很类似于写js中用到最多的console.log方法，但是不同于console.log你不需要在生产环境下去全局搜索，然后再注释掉这些console.log，你只需要在开发的时候打开debug功能，在上线的时候不打开就行了。

在Express开发中，如果你需要看到所有的内部的log信息，只需要一行

DEBUG=express:* node index.js

通过设置DEBUG环境变量为express:*就行了；

Express进程管理

当在生产环境使用Express的时候，使用一个进程管理来监听Express的进程是非常有必要的。简单的带来以下好处：

  • 自动重启崩溃的Express进程；
  • 能够监听运行时Express进程的性能和资源消耗状况
  • 动态的修改配置以提高应用性能
  • 集群控制

一个进程管理类似于一个应用Server能够作为一个应用的容器用来快速部署，提供高可用，并且让你能够在应用运行时管理应用；

常用的一些进程管理工具：PM2，forever；

PM2

自带负载均衡，这个有点厉害；

能够快速的帮你通过cluster的模式启动四个进程。还能通过pm2 scale [app-name] 10的方式快速扩展你的cluster到10个进程；
